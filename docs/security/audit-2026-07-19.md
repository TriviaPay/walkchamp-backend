**Top themes**
- **Payment integrity** dominates: entry-fee escrow gaps, coin-ledger double-spend, withdrawal TOCTOU, prize/refund payout weaknesses (PAY-06, PAY-07, PAY-09, PAY-11, PAY-17, PAY-18, PAY-19).

- **Weak crypto + enumeration**: non-CSPRNG invite/referral codes with no rate limiting on join-by-code (CRYPTO-01).

## High Severity

### H-1 · Coins-battle free-ride — participant kept in race after paying zero/partial entry
**Location:** `src/routes/races.ts:2629-2630` (also `src/lib/coinsBattle.ts:193-201`)
**Exploit:** Join does only a soft balance check with no escrow. User joins with enough coins, spends them elsewhere before start. At start, `races.ts:2629` deducts `Math.min(currentBalance, coinEntryAmount)`; when balance is 0, line 2630 `if (toDeduct <= 0) continue;` skips the charge but leaves the participant in the race, eligible for a prize pool funded by others' full entries. Reliable and repeatable.
**Fix:** Escrow `coinEntryAmount` atomically at join (row-lock wallet, deduct into held balance), or at start disqualify anyone who can't pay the full entry instead of `continue`-ing. No prize eligibility unless the full entry was collected.

### H-2 · Race reconciliation overwrites server steps with loosely-capped client value and auto-verifies
**Location:** `src/routes/races.ts:4914-4917`
**Exploit:** Live sync enforces a strict per-second delta cap (server is authoritative), but reconciliation accepts a fresh client step count bounded only by `maxPlausibleSteps = ceil(raceDurationSeconds * 10)` and never cross-checks the server's stored progress. A finisher POSTs `source='healthkit'` with steps near the ceiling; it passes the `>existingSteps`/`<=maxPlausibleSteps` guards, overwrites `race_results.steps`, and unconditionally sets `status='verified'` — altering rank/prize eligibility in cash races.
**Fix:** Cap accepted steps at the server's last-synced `currentSteps` (+ small tolerance), not a duration-derived ceiling. Derive `verified` from server evidence, not the client `source` field.

### H-3 · Withdrawal approval TOCTOU — double approval / double payout
**Location:** `src/routes/admin.ts:628-640`
**Exploit:** Reads the row, checks `status !== 'pending'`, then `UPDATE ... SET status='approved' WHERE id=?` with no transaction, no `FOR UPDATE`, no compare-and-swap. Two concurrent approvals (two admins, or a double-click/retry) both pass and both commit → withdrawal paid twice. The operational lock is a coarse feature flag, not a row guard.
**Fix:** Atomic transition `UPDATE ... SET status='approved' WHERE id=? AND status='pending'`; if `rowsAffected=0` return 409. Wrap read+checks+update in a transaction with `SELECT ... FOR UPDATE`; add an idempotency key on the payout call.

### H-4 · `recordCoinLedgerEntry` read-then-write without row lock — coin double-spend
**Location:** `src/lib/coinsService.ts:114-155`
**Exploit:** Plain `SELECT` of `currentBalance` (no `FOR UPDATE`), checks `+amount >= 0`, then writes an absolute `newBalance`. Idempotency key only dedups identical transactions. Two distinct concurrent spends read the same balance, both pass, both commit — last-writer-wins loses one debit. 100 coins → spend 100+100 concurrently, get both goods.
**Fix:** `SELECT ... FOR UPDATE` in a transaction, or atomic `UPDATE ... SET currentBalance = currentBalance + :amount WHERE userId=? AND currentBalance + :amount >= 0` checking `rowsAffected`; add DB `CHECK (currentBalance >= 0)`.

### H-5 · Host removal from paid rooms issues no refund, hardcodes `refundProcessed:false`
**Location:** `src/routes/races.ts:5115-5132`
**Exploit:** Host removes a participant from a paid open/full/scheduled room; handler only flips status to `left` and decrements `currentPlayers` — no wallet credit/reversal, and Pusher (5154-5155) + HTTP (5164-5165) hardcode `refundProcessed:false`/`refundAmount:0`. Removed player loses their fee; a colluding host farms forfeited stakes into a pool he wins.
**Fix:** In the removal transaction, look up the fee paid by `targetUserId` and credit/reverse idempotently in the same transaction; compute the refund fields from the real result; guard double-refund.

### H-6 · Mass-assignment — user self-sets `emailVerified:true`, backdates login timestamps
**Location:** `src/routes/auth.ts:398-404`
**Exploit:** PATCH profile with `{"emailVerified":true}`. `updateProfileSchema` accepts it and the handler writes it (line 422) with no verification flow — bypasses any trust gate keyed on `emailVerified`. `lastLoginAt`/`lastSeenAt` (425-426) can be backdated to forge history.
**Fix:** Remove `emailVerified`, `lastLoginAt`, `lastSeenAt` from the client schema. Set `emailVerified` only via the server-side Descope/OTP callback; set timestamps server-side. Whitelist mutable fields (`avatarColor`, `bio`).

### H-7 · Backdated `localDate` enables retroactive milestone coin farming
**Location:** `src/routes/walk.ts:59-69`
**Exploit:** `POST /walk/steps` accepts any valid `localDate` with no range check. Milestone dedup is date-scoped (`daily_walk_2020-01-01`) with `ON CONFLICT DO NOTHING`, so submitting deltas for hundreds of unclaimed past dates makes `evaluateStepMilestones` (429) award coins for each — coin farming at scale.
**Fix:** Reject `localDate` outside today/yesterday (user tz) with 400; skip milestone evaluation for backdated submissions; add a per-user/day cap on cumulative delta steps.

### H-8 · Low-entropy invite/referral codes + no rate limit on join-by-code
**Location:** `src/routes/groups.ts:69-71` (also `races.ts:2334-2337`, `auth.ts:332/512`)
**Exploit:** `makeInviteCode()` uses `Math.random()` (~31 bits, recoverable PRNG state); race codes use `randomBytes(5)` mapped via `b%32` (modulo bias → ~25 bits / ~33M codes); referral codes also `Math.random()`. No join-by-code endpoint is rate-limited. Attacker enumerates the space to join private/cash rooms or predict referral codes.
**Fix:** CSPRNG (`crypto.randomBytes`) with rejection sampling, ≥64 bits entropy; rate-limit + lockout on all join/lookup-by-code endpoints; add code expiry; replace every `Math.random()` used for security tokens.

## Medium Severity

### M-1 · Prize payout idempotency keyed on rank permits cross-rank double credit
**Location:** `src/lib/cashChallengePayments.ts:78`
**Exploit:** Key is `prize:{raceRoomId}:{userId}:{rank}` guarded only by `onConflictDoNothing`. If the same `(raceRoomId,userId)` is processed with a different rank (re-rank after DQ, or a user listed twice in payouts) a new key is produced and the credit runs again. No cross-rank existence check like the entry side's `hasCompletedEntryPayment`.
**Fix:** Check for any existing completed prize for `(raceRoomId,userId)` regardless of rank; add a unique constraint on `(raceRoomId,userId,transactionType='race_prize_paid')`; dedupe `userId` in the payouts array.

### M-2 · `SPECTATE_MATCH` reward farmable via multiple spectate sessions
**Location:** `src/routes/spectate.ts:53-59`
**Exploit:** `POST /spectate/start` can be called repeatedly for one race (no unique `(userId,raceRoomId)`), each creating a fresh `sessionId`. `grantCoinReward` idempotency is keyed on `sourceId=sessionId`, so each session yields +2 coins; the 60s gate is per-session and parallelizable → unlimited farming per race.
**Fix:** Unique constraint on `spectateSessionsTable(userId,raceRoomId)`, or use `sourceId=raceRoomId` in `grantCoinReward` so the existing unique index caps it once per race.

### M-3 · No membership check before Pusher broadcast / comment insert on caller-supplied IDs
**Location:** `src/routes/chat.ts:451-472`
**Exploit:** `POST/DELETE /api/chat/private/react` with an arbitrary `conversationId` triggers `chat:reactions_updated` on `private-chat-{conversationId}` for conversations the caller isn't in. `POST /api/races/:id/comments` has no participant check — any user inserts comments and broadcasts `race:comment_added` on arbitrary races.
**Fix:** Verify the user is a participant of the conversation/race before any DB mutation or Pusher trigger; validate `messageId` belongs to `conversationId`; 403 otherwise.

### M-4 · Unauthenticated deposit browser-cancel/fail writes unbounded reason into any user's transaction metadata
**Location:** `src/routes/deposit.ts:967-1000`
**Exploit:** `browser-cancel`/`browser-fail` are unauthenticated. `markDepositReturnObserved` resolves the transaction by UUID with no owner check and merges an uncapped `reason` into metadata JSONB. Anyone with a leaked deposit UUID overwrites metadata with huge/arbitrary strings, corrupting reconciliation; no rate limit. Only v4 UUID unpredictability protects it.
**Fix:** Bind the callback to the payer (authenticated session or a short-lived signed token from deposit creation threaded through the return URL); cap `reason` (~200 chars); rate-limit; make the patch idempotent.

### M-5 · `validateThemeOwnership` fails open (returns `true`) on any thrown error
**Location:** `src/routes/trackThemes.ts:512-514`
**Exploit:** The ownership check's `catch` block unconditionally `return true`. Any error in the lookups (496-509) — pool exhaustion, statement timeout, deadlock, inducible by flooding race creation — grants access, letting a non-owner use a paid theme (paywall bypass). Auth control fails open.
**Fix:** `catch` → log and `return false` (deny); propagate the error so creation aborts 5xx.

### M-6 · Group step sync — no upper bound, no date range, no rate limit
**Location:** `src/routes/groups.ts:1083-1090`
**Exploit:** `stepSyncSchema` bounds `dailySteps` only with `min(0)` and accepts any `stepDate` (incl. future), no rate limiter. An active member upserts arbitrarily large counts for any date repeatedly, poisoning group leaderboards/history.
**Fix:** Cap `dailySteps`/`verifiedSteps` (e.g. 200,000 like `walk.ts`), restrict `stepDate` to a recent window (no future), add a rate limiter; prefer verified counts.

## Low Severity

### L-1 · Refund webhook amount binding skipped for non-numeric amount
**Location:** `src/lib/refundService.ts:1078`
**Exploit:** `typeof eventAmount === "number"` silently skips amount verification when the provider sends a string/omits it. Impact bounded: DB update uses stored `item.approvedAmount` (no over-credit), currency/paymentId still checked, signature verified upstream. Residual: weakened event→refund binding could bind a mismatched-amount event.
**Fix:** `const amt = Number(eventAmount); if (!Number.isInteger(amt) || amt !== item.approvedAmount)` → treat as mismatch/`requires_review`.

### L-2 · `onConflictDoUpdate` reassigns a OneSignal device to the caller with no prior-ownership check
**Location:** `src/routes/push.ts:61-74`
**Exploit:** If an attacker learns a victim's `onesignalPlayerId`, upserting it overwrites `userId` to the attacker — victim's device now receives attacker's pushes. Bounded because OneSignal IDs are opaque client-generated UUIDs not exposed to others.
**Fix:** On conflict, reject when the existing `userId` differs (409), or require proof of device control (OneSignal external identity token bound to the user) before transfer.

## Remediation status (2026-07-20)

All 16 findings fixed on branch `security/audit-2026-07-fixes` (code + regression tests; `pnpm typecheck` and `pnpm test` green). See `docs/security/audit-2026-07-19-fixes.md` for the per-finding change list.

### Deferred DB hardening (defense-in-depth, requires data cleanup first)

These back up code fixes that already close the findings. Deferred because each can fail `db:migrate` against existing data created by the original bugs — verify/clean first, then add to the schema and `pnpm db:generate`:

- `CHECK (current_balance >= 0)` on `coin_balances` (H-4) — will fail if the double-spend bug left any negative balances; audit balances first.
- Partial unique index on `wallet_transactions (race_room_id, user_id) WHERE transaction_type='race_prize_paid'` (M-1) — will fail if duplicate prize rows already exist.
- Spectate: intentionally **not** adding a `unique(user_id, race_room_id)` on `spectate_sessions` — it would block the legitimate leave-and-re-watch flow. The reward farming is closed at the code level (reward keyed on `raceRoomId`).
